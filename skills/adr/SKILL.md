# ADR スキル

リポジトリの設計判断を ADR (Architecture Decision Record) として構造化・記録するスキル。

## 概要

ADR はソフトウェアプロジェクトにおける重要な設計判断とその背景を記録するドキュメント。
「なぜそうしたか」を残すことで、将来の開発者（未来の自分を含む）が判断の経緯を理解できる。

## ワークフロー

### ステップ 1: ソースの特定と読み込み

ADR の根拠となるソースをユーザーに確認し、読み込む。

**ソースの種類**（複数の組み合わせ可）:

| ソース | 説明 | 読み込み方法 |
|--------|------|-------------|
| ソースコード | リポジトリの主要ファイル | 実装と構成を直接読む |
| 既存ドキュメント | README.md, CLAUDE.md, AGENT.md, docs/ 等 | 直接読む |
| ユーザーの口頭説明 | 会話で伝えられた設計判断 | 会話コンテキストから抽出 |
| Git 履歴 | コミットメッセージや PR 説明 | `git log` で参照 |

**ソース読み込みのポイント**:
- まず実装と既存ドキュメントを優先する
- ソースコードから、使用ライブラリ・アーキテクチャパターン・モジュール構成を読み取る
- 判断材料が不足する場合だけユーザーに補足を確認する

### ステップ 2: 設計判断の抽出

ソースから「決定」と呼べるものを洗い出す。以下の観点で判断を特定する:

- **技術選定**: なぜこのライブラリ/フレームワーク/言語を選んだか
- **アーキテクチャ**: モジュール構成、データフロー、API 設計
- **トレードオフ**: 何かを得るために何を諦めたか
- **制約への対応**: 外部制約（API 制限、パフォーマンス要件等）にどう対処したか
- **方針転換**: 当初の計画を変更した理由

抽出した判断が複数ある場合は候補を短く列挙し、今回 ADR 化する対象を絞る。

### ステップ 3: テンプレート選択

`templates/` ディレクトリからタスクに適したテンプレートを選択する。

現在利用可能なテンプレート:

| テンプレート | ファイル | 用途 |
|-------------|---------|------|
| 標準 ADR | `architecture-decision.md` | 一般的な設計判断の記録 |

テンプレートを Read で読み込み、プレースホルダーを具体的な内容に置き換える。

### ステップ 4: ADR ファイルの作成

以下のルールに従ってファイルを作成する:

**ファイル命名規則**:
```
ADR-{連番3桁}-{kebab-case-の簡潔なタイトル}.md
```
例: `ADR-001-monorepo-structure.md`

**配置先**: ユーザーが指定したディレクトリ（デフォルト: `docs/adr/`）

**連番の決定**:
- 既存の ADR がある場合、最大番号の次を採番
- ない場合は 001 から開始

**記述のポイント**:
- 「コンテキスト」は当時の状況を再現できるよう、背景と制約を具体的に書く
- 「決定」は実装の詳細ではなく、判断そのものを簡潔に
- 「理由」は代替案との比較を含め、なぜその選択が最善だったかを説明
- 「影響」は良い面も悪い面も正直に書く。将来の注意点や制約も含める
- 技術用語の説明は、チーム外の人が読む可能性を考慮して判断する

### ステップ 5: インデックスの作成・更新

ADR ディレクトリの `README.md` をインデックスとして作成または更新する。

**新規作成の場合**: テーブル形式で全 ADR をリスト
**既存更新の場合**: 新しい ADR の行を追加

インデックスのフォーマット:

```markdown
# Architecture Decision Records (ADR)

| ADR | タイトル | 対象 | ステータス |
|-----|---------|------|-----------|
| [ADR-001](ADR-001-xxx.md) | タイトル | 対象コンポーネント | Accepted |
```

### ステップ 6: 確認

作成した ADR のパス、要約、追加または更新したインデックスをユーザーに報告する。

## テンプレートの追加

新しいテンプレートを追加する場合:

1. `templates/` に新しい `.md` ファイルを作成
2. ファイル先頭に YAML frontmatter でメタデータを記述（`template_name`, `description`, `sections`）
3. プレースホルダーは `{{placeholder_name}}` 形式で記述
4. このファイル（SKILL.md）のステップ 3 のテンプレート一覧テーブルを更新
