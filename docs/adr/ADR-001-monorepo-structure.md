# ADR-001: モノレポ構成（lol_review + lol_vod_analyzer）

## ステータス

Accepted

## コンテキスト

LoL 上達支援のために、2つの異なる分析アプローチが必要だった:

1. **試合データ分析** — Riot API から統計データを取得し、プレイの傾向を数値で把握する
2. **動画分析** — 動画を AI で分析し、言語化されたナレッジを抽出する
   - **解説動画分析**: YouTube の解説動画から知識や判断基準を抽出する
   - **プレイ動画分析**: 自分の録画やリプレイ動画を振り返る

加えて、`試合データ付きリプレイ分析` は独立した第3の分析系統ではなく、**プレイ動画分析に試合データを追加する拡張ワークフロー**として扱う。

これらを単一リポジトリで管理するか、別リポジトリにするかの判断が必要だった。

## 決定

**単一リポジトリ（モノレポ）に2つの独立したPythonパッケージを配置する。**

```
lol-tools/
├── lol_review/          # 試合データ分析
│   ├── src/
│   ├── tests/
│   └── pyproject.toml
└── lol_vod_analyzer/    # 動画分析
    ├── src/
    ├── tests/
    └── pyproject.toml
```

各パッケージは独立した `pyproject.toml` と `uv` 仮想環境を持ち、依存関係は分離されている。

## 理由

- **データフロー連携**: `lol_review` の出力（`latest_findings.json`）を `lol_vod_analyzer` の `--match-data` オプションで読み込むパイプラインがある。同一リポジトリ内であれば相対パスで容易に連携できる
- **開発効率**: 両プロジェクトは同一ドメイン（LoL 分析）に属し、同時に開発・改善するケースが多い
- **依存関係の分離**: `uv` のプロジェクト単位管理により、パッケージ間の依存汚染は発生しない。lol_review は httpx + Riot API、lol_vod_analyzer は google-genai + OpenCV + yt-dlp と全く異なる依存を持つ
- **個人プロジェクト**: チーム開発ではないため、リポジトリ分割によるアクセス制御やCI分離のメリットが薄い

## 影響

- 各パッケージのセットアップ・テストは個別に実行する必要がある（`cd lol_review && uv sync`）
- CI を導入する場合、変更検知でパッケージごとにジョブを分ける設計が必要
- 将来パッケージが増えた場合は、ワークスペース管理ツール（uv workspace 等）の導入を検討する
