# ADR-014: 重要シーン優先のスクリーンショット選定

## ステータス

Accepted

## コンテキスト

ADR-012 で adaptive interval と momentum impact を導入したが、スクリーンショット抽出自体は依然として「動画全体を刻む」発想が強く、長時間動画では出力量が動画長に比例しやすかった。

Issue #18 では、Windows 環境の replay / VOD 分析で `packages/lol_vod_analyzer/output/screenshots` に数十万ファイル規模の画像が生成され、ストレージを大きく圧迫した。cleanup だけでは「取りすぎる」問題そのものは残る。

また、分析観点でも全時間帯を均等に撮ることは最適ではなかった。

- 試合開始直後の数分は、初動ルートやレベル 1-3 の判断を拾いたい
- `match_data` がある場合は、モーメンタムの大きい区間をより密に見たい
- HTML レポートには分析に使った全画像ではなく、代表画像だけあれば十分

### 代替案

1. **interval のデフォルトを広げる**: 出力量は減るが、依然として動画長に比例し、重要シーンの密度制御もできないため却下
2. **cleanup のみ追加する**: 実行後のストレージは抑えられるが、分析コストと抽出処理の無駄が残るため却下
3. **全画像を HTML に埋め込み続ける**: 画像ファイルの代わりに HTML が肥大化するだけなので却下
4. **完全にモーメンタム依存で抽出する**: `match_data` がない動画で使えず、静かながら重要な序盤文脈も落とすため却下

## 決定

**スクリーンショット抽出を「全時間の等間隔サンプリング」から「固定予算内の重要シーン優先サンプリング」に変更する。**

具体的には:

- `max_screenshots` を導入し、分析に使うスクリーンショット総数の上限を固定する
- gameplay 文脈では、試合開始直後の early-game 枠を予約する
- `match_data` がある場合は、`gold_diff_timeline` から得た momentum 高区間を優先候補にする
- その上で adaptive interval または fixed interval を補完枠として使う
- HTML レポートには代表画像だけを残し、分析用の全画像は表示しない
- デフォルトでは run-specific な一時ディレクトリに保存し、レポート生成後に削除する
- 明示的に残したい場合だけ `--keep-screenshots` で保持する

## 理由

- **分析品質とストレージ制御を同時に改善できる**: 重要時間帯へ予算を寄せるほうが、単なる cleanup より本質的
- **既存の momentum ロジックを再利用できる**: match-data 圧縮に使っている重要時間帯をスクリーンショット選定にも流用できる
- **early-game の文脈を守れる**: モーメンタムだけでは拾いにくい序盤を固定枠で確保できる
- **後方互換な拡張にしやすい**: `--max-screenshots` と `--keep-screenshots` を追加するだけで CLI に無理なく露出できる
- **表示責務を増やしすぎない**: `AnalysisResult.snapshots` を代表画像集合として使えば、レポート側の変更を最小化できる

## 影響

- gameplay / replay 分析では、画像抽出が「全体を均等に見る」より「重要時間帯を見る」性質に変わる
- `match_data` がない通常動画では、momentum 強化なしで adaptive/fixed の補完抽出を使う
- HTML に表示される画像数は減るが、レポートの可読性とサイズは改善する
- `--keep-screenshots` 指定時は分析用の中間画像も保持される。代表画像だけを残す専用モードは今後の拡張候補
