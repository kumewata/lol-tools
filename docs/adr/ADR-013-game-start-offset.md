# ADR-013: 動画上の試合開始オフセットによるタイムライン同期

## ステータス

Accepted

## コンテキスト

ADR-005 で導入した match-data 連携では、Riot API のイベントタイムスタンプ（試合開始=0秒基準）を動画のチャンク時間範囲と直接比較していた。これは「動画の0秒地点 = 試合の0秒地点」という暗黙の前提に依存していた。

しかし実際の動画では以下のようなケースがある:

- リプレイ録画にロード画面やチャンピオン選択画面が含まれている
- 配信 VOD で試合前のトーク・待機時間がある
- 録画開始が試合開始より遅れた（まれなケース）

このずれにより、`_build_chunk_timeline` が誤った時間範囲のイベントをチャンクに割り当ててしまい、LLM が画面と無関係なイベントデータを参照する問題があった。

### 代替案

1. **動画のスクリーンショットから試合開始を自動検出**: LLM やOCR でゲーム内タイマーの「0:00」を探す方法。精度・コスト・実装複雑度の面で却下
2. **match-data 側にオフセットを埋め込む**: JSON に `video_offset` フィールドを追加する方法。データ生成側（lol_review）とデータ消費側（lol_vod_analyzer）の結合度が上がるため却下
3. **CLI オプションでユーザーが指定**: シンプルかつ確実。ユーザーは動画を数秒見れば試合開始時刻を特定できる

## 決定

**`--game-start <秒>` CLI オプションを追加し、動画上の試合開始時刻をユーザーが指定する。**

変換式: `game_time = video_time - game_start_offset`

適用箇所:
- `build_gameplay_image_prompt` → `_build_chunk_timeline` 呼び出し時にチャンク時間範囲をゲーム時間に変換
- LLM プロンプト内の時間表記をゲーム時間で表示

適用しない箇所:
- スクリーンショットの `timestamp_ms`（動画基準のまま。レポートのタイムスタンプリンクに使用）
- 字幕セグメントの `start_ms` / `end_ms`（動画基準のまま）

## 理由

- **シンプルさ**: 変換は引き算1つ。既存のデータモデルやファイルフォーマットに変更不要
- **確実性**: 自動検出と異なり、ユーザーが目視で確認した値を使うため誤差がない
- **既存パターンとの一貫性**: `--speed` オプションと同様、動画とゲーム時間の変換を CLI オプションで制御するパターンを踏襲
- **デフォルト値 0 による後方互換性**: オプション未指定時は従来通り動作

## 影響

- match-data 連携時にユーザーが動画の試合開始時刻を把握する必要がある（多くの場合は自明）
- 負のゲーム時間（オフセット以前の動画区間）のチャンクではイベントが空になるが、これは正しい挙動
- 将来、自動検出（ゲーム内タイマー OCR 等）を実装する場合、このオプションのデフォルト値を自動検出結果に置き換えるだけで対応可能
