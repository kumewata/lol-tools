# ADR-009: 初回ユーザー導線の統一

## ステータス

Accepted

## コンテキスト

`lol-tools` は試合データ分析と動画分析を同一リポジトリで扱うモノレポであり、内部的には `lol_review` と `lol_vod_analyzer` の 2 つのパッケージから構成される。一方で、リポジトリ直下には統一 CLI として `lol-tools` が存在し、新規ユーザーに対しては単一の入口を提示できる状態になっている。

しかし、現時点の利用導線には次のズレがある。

- README が旧構成前提の説明を含み、`cd lol_review` や `cd lol_vod_analyzer` を前提にしている
- `.env` の設定先の説明が現在の実装と一致していない
- 初回セットアップで必要な確認事項が分散しており、何から試せばよいかが分かりにくい
- ユーザーが最初に知りたい情報は「内部構成」ではなく「最短で動かす方法」だが、現状の README はその順序になっていない

この状態では、リポジトリを clone したばかりのユーザーが、正しい起動方法に到達する前に構造理解を強いられる。

## 決定

新規ユーザー向けの導線は、内部パッケージ構成ではなく「やりたいこと」から始まる単一入口として設計する。

具体的には、以下を採用する。

1. 実行入口を `uv run lol-tools ...` に統一し、README の表層から `packages/` や個別パッケージのディレクトリ移動を隠す
2. README の先頭をクイックスタート中心に再構成し、最初の 5 分で試せるコマンドを最優先で提示する
3. 初回セットアップ専用の補助コマンドを追加する
4. エラーになる前に不足設定を把握できる診断導線を追加する
5. 詳細情報は段階的に開示し、最初のページに内部事情を並べない

初回ユーザーに提示する標準フローは次のとおりとする。

```bash
git clone <repo>
cd lol-tools
uv sync
cp .env.example .env
uv run lol-tools init
uv run lol-tools doctor
uv run lol-tools review
```

動画分析は同じ入口配下で次の形式に統一する。

```bash
uv run lol-tools vod analyze 'https://youtube.com/watch?v=...'
uv run lol-tools vod analyze ~/Desktop/replay.mov --mode gameplay
```

README は次の順序に並べ替える。

1. このリポジトリでできること
2. 3 分クイックスタート
3. 目的別の最短コマンド
4. リプレイ分析の標準ワークフロー
5. トラブルシュート
6. 開発者向け情報

補助コマンドの役割は次のように定義する。

- `lol-tools init`
  - `.env` がなければ雛形を作成する
  - `RIOT_API_KEY`、`GOOGLE_API_KEY`、`DEFAULT_RIOT_ID`、`DEFAULT_COUNT` の設定を支援する
  - 対話式でも非対話式でも使えるようにする
- `lol-tools doctor`
  - `.env` の有無
  - 必須 API キーの有無
  - `ffmpeg` の有無
  - よく使うサブコマンドが実行可能か
  - 失敗時の具体的な修正手順
    を表示する
- `lol-tools examples`
  - 代表的な実行例だけを短く一覧表示する

## 理由

- 新規ユーザーは実装構造ではなく「最初の成功体験」を求めている。入口を 1 つに絞ることで探索コストを減らせる
- 現在の実装はすでにリポジトリルートの `.env` と統一 CLI を前提にしており、README をその実装に揃える方が自然である
- `init` と `doctor` を用意すると、README にすべての注意点を書き込まなくても自己診断で不足を埋められる
- 「クイックスタート -> 目的別例 -> 詳細」の順に情報を並べると、初見ユーザーと開発者の両方に対応しやすい
- モノレポ内部のパッケージ境界は保ちながら、利用時の認知負荷だけを下げられる

代替案も検討した。

- 現状維持で README だけ少し修正する案
  - 最低限の改善にはなるが、設定漏れや依存漏れを実行前に発見できない
- パッケージ単位の CLI を正規入口として残す案
  - 内部構造の理解を要求するため、新規ユーザー向けの導線としては弱い
- GUI や Web UI を先に作る案
  - 将来的な選択肢にはなるが、現時点では CLI とドキュメントを整える方が変更量と効果の釣り合いが良い

## 影響

良い影響:

- clone 直後のユーザーが README と CLI ヘルプだけで最初の実行に到達しやすくなる
- README、`.env.example`、CLI 実装の前提が一致する
- サポート時に案内するコマンドを `lol-tools` 配下へ統一できる
- 将来サブツールが増えても、入口設計を維持したまま拡張できる

注意点:

- `init` と `doctor` を追加する場合、CLI 層に設定・診断ロジックを集約する実装が必要になる
- 既存ユーザー向けに、旧 README のコマンドから新導線への移行を明記する必要がある
- ADR-001 の例示パスや README 内の旧ディレクトリ構成の記述は、現行構成に合わせて別途更新が必要になる

次の実装単位は以下を推奨する。

1. README を新導線に合わせて全面改稿する
2. `lol-tools init` を追加する
3. `lol-tools doctor` を追加する
4. `lol-tools examples` とヘルプ文の改善を行う
